### Задание 1

```
1.1. Бюджетирование проектов, финансовые отчёты и прогнозирование рисков
-Тип СУБД: Реляционная (SQL) СУБД, например, PostgreSQL или Microsoft SQL Server.

1.1. Ускорение хеширования*
-Решение: Использовать API для кэширования, например, Redis API. Redis хранит данные в оперативной памяти, что ускоряет операции хеширования и частые запросы.

1.2. Лендинги и CRM
-Для лендингов: NoSQL СУБД, например, MongoDB. Причина: Гибкость схемы, быстрое чтение/запись, возможность хранить данные о лидах в формате JSON.

-Для CRM: Реляционная СУБД, например, PostgreSQL. Причина: Удобство управления связями между клиентами, сделки, история взаимодействий.

1.2. Можно ли закрыть задачу одной СУБД?*
-Да, можно использовать PostgreSQL с поддержкой JSONB. Это сочетает реляционную структуру для CRM и гибкость NoSQL для лендингов.

1.3. База знаний для отдела контроля качества
-Тип СУБД: Документная СУБД, например, MongoDB.
-Причина: Простота хранения иерархических данных (нормы, правила, материалы) в виде документов.

1.3. Использование существующей СУБД*
-Да, можно использовать PostgreSQL с таблицами для хранения структурированных данных и JSONB для гибких материалов. Это унифицирует архитектуру.

1.4. Логистика и маршруты доставки
-Тип СУБД: Графовая СУБД, например, Neo4j.
-Причина: Эффективная работа со связями (маршруты, узлы, курьеры), быстрое построение оптимальных путей.

1.4. Подключение отдела закупок*
-Лучше использовать отдельную СУБД, но интегрированную с графовой СУБД логистов. Например, PostgreSQL для закупок (учет материалов) и Neo4j для маршрутов. Это обеспечит гибкость и производительность.

1.5. Решение всех задач одной СУБД*
Да, можно использовать PostgreSQL.

Реализация:
-Для финансов (1.1) — таблицы с транзакциями.
-Для CRM и лендингов (1.2) — таблицы и JSONB.
-Для базы знаний (1.3) — таблицы и документы JSONB.
-Для логистики (1.4) — расширение Apache AGE для работы с графами.
```

```
Задание 3. SQL vs NoSQL
3.1. Преимущества SQL перед NoSQL
1. Целостность данных (ACID).

2. Стандартизированный язык (SQL).

3. Сложные запросы (JOIN, агрегации).

4. Транзакционность.

5. Надёжность и зрелость технологий.

3.1. Преимущества NewSQL*
1. Масштабируемость как у NoSQL.

2. Поддержка ACID как у SQL.

3. Высокая производительность в распределённых средах.
Пример: Google Spanner, CockroachDB.

### Задание 4

```
Задание 4. Кластеры
Критерии выбора СУБД
Масштабируемость — возможность горизонтального масштабирования.

Поддержка распределённых вычислений — встроенные механизмы для работы с большими данными.

Отказоустойчивость — репликация и шардирование.

Производительность — оптимизация для параллельных вычислений.

Модель распределённых вычислений
MapReduce (например, в Hadoop) или Apache Spark.

Причина: Обработка больших объемов данных на множестве машин, устойчивость к сбоям, параллельное выполнение задач.

Пример СУБД: ClickHouse для аналитики или Cassandra для записи.
